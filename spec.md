# XUTION Portal

## Current State
QR login decodes a QR from a camera scan or uploaded image on the login screen, then matches the decoded data against:
- The raw username (plain text)
- JSON with `username` field
- A generated `xution_qr_token` stored in member extras on the backend

When L6 uploads an ID card image for a member in the Sovereign Database, the image is stored in localStorage but the QR code embedded in that card is NOT extracted or stored.

This means QR login only works if the QR was generated by the system (xution_qr_token) or if the QR plainly encodes the username — it doesn't support matching against QR codes embedded in uploaded full ID card images.

## Requested Changes (Diff)

### Add
- A helper function `extractQRFromImage(dataUrl: string): Promise<string | null>` that loads jsQR and decodes a QR code from an image (handles full card images, not just standalone QR images)
- When L6 imports/uploads an ID card image for a member in the Sovereign Database, automatically call `extractQRFromImage` on the uploaded image, and if a QR is found, store the decoded QR data in the member's backend extras as `idCardQrData`
- On QR login (both camera scan and uploaded image paths), after decoding the QR from the presented card, also try matching the raw decoded string against `idCardQrData` stored in all members' extras via `actor.getAllMemberExtras()`

### Modify
- The `onScan` handler in `QRLoginScanner` / login flow: after the existing username/token matching logic, add a new fallback step that checks if the decoded QR string matches any member's stored `idCardQrData` in backend extras, and if so, logs in as that member
- The ID card import handler in the Sovereign Database section: after setting the card image, call `extractQRFromImage` on the uploaded image and store the decoded QR data in member extras via `actor.setMemberExtras`

### Remove
- Nothing removed
